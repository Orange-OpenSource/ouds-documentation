Definition

## What is an agentic design system?

An agentic design system is infrastructure that lets AI agents autonomously read, reason over and build with your components, tokens and guidelines. Unlike traditional systems written for humans, it encodes intent, relationships and constraints as machine-readable context, so agents can observe, detect, suggest, fix and learn inside a self-healing loop with human oversight.

## Key concepts

### Design system as infrastructure

Think of your design system less like a Figma library and more like a CI/CD pipeline or database. It becomes the API that lets AI agents build your product safely.

### Observe, detect, suggest, fix, learn

A self-healing loop based on IBM's 2003 MAPE-K framework. Signals from Figma API, CI hooks and usage analytics flow into a drift-scoring engine that creates PRs automatically.

### AI is a new user

Your design system was written for humans. The new user, AI, needs structured metadata, not documentation prose. Components need to be machine-readable.

### MCP is deterministic, LLM is stochastic

The same MCP query returns the same data, but the LLM reasons differently each time. Structured data like JSON mitigates that randomness.

### Context > Probability

Without infrastructure, AI collapses toward the average of the internet. Design systems give AI something it cannot get from training data: your encoded decisions.

### Trust levels, not full autonomy

Agents earn trust like career progression. Start as an intern (suggest only), then junior (mechanical fixes), then more autonomy. Some decisions will always need human judgment.

### Plant seeds, not trees

Foundations before automation. Naming conventions, token structure and component descriptions first, then build agents on top.

### Progressive disclosure of context

Always-on rules for foundations (spacing, color, typography), MCP on-demand for components, AGENTS.md as orchestration layer. Together they form a plugin.

## How to start

Seven practical steps from the speakers to make your design system agent-ready, starting Monday.

01

### Plant seeds, not trees

Start with foundations: naming conventions, token structure, component descriptions. Don't chase full autonomy on day one.

Romina Kavcic

02

### Make components machine-readable

Codify knowledge as structured metadata, not documentation prose. Start with a JSON schema per component: identity, props, variants, sizes, states, usage.

Diana Wolosin, Indeed

03

### Add intent to Figma descriptions

Write what the component does and how it can be combined, not just specs. Agents need intent, not just visuals.

Romina Kavcic

04

### Set trust levels per action

Decide whether an agent can auto-merge, draft a PR, or only suggest. Low-risk and high-confidence = auto. Everything else = human review.

Romina Kavcic

05

### Colocate and name semantically

Keep component rules in one place. Use semantic token names like danger.background, not raw red-500. Add front-matter to docs so they're parseable.

Jan Six, GitHub

06

### Benchmark before committing to a format

Don't plug your Markdown into an MCP and call it done. Test JSON vs MD vs hybrid against accuracy and token cost on real queries.

Diana Wolosin, Indeed

07

### Pair with a developer

Not just weekly syncs. Real pair programming. This is what's missing in most teams building agent-ready systems.

Romina Kavcic
