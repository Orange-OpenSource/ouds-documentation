## Designers obsess over component libraries. The real system agents work with is how context flows — and most design teams haven't built that yet.

==**In this article:**==

- Why a complete-looking design system is still invisible to an AI agent
- What the Context Gap is, and how to measure whether your system has one
- The design systems skill that's in demand in 2026 and still going mostly unnamed

You spent six months building it—every component is named. Every token is mapped. Every interaction state is documented. The Figma file was the kind that made new engineers stop scrolling and actually read the annotations.

Then you connected [Claude Code](https://claude.ai/code) to the [Figma MCP](https://www.figma.com/blog/the-figma-canvas-is-now-open-to-agents/). The agent started generating screens.

It didn't use a single component.

Most design teams think the agentic AI problem is a component problem: if you structure your Figma components correctly, the agent will use them. That assumption is wrong. The problem is context — specifically, the absence of it. And most design teams haven't started solving it yet.

The gap between a design system that looks complete to a human reviewer and one that an AI agent can actually use — I call this the Context Gap. It's the gap this article is about.

## What agents actually need

![A component card split into two readings — what a human designer sees (full visual render) versus what an AI agent sees (name only, with empty usage rule fields marked by lime green question marks).](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*uo4EyqY03J-yccNb2N9lwg.png)

illustration by author | Generated by Gemini

When a designer opens a Figma file, they see a visual composition. They read component names, recognize spatial relationships, and absorb the system's logic through pattern recognition built over years of practice.

An agent reads none of that the same way.

[Christine Vallaure](https://uxdesign.cc/agentic-ai-design-systems-figma-a-practical-guide-6ab0b681718d), writing about agentic AI and design systems in March 2026, named it directly: "The design system is no longer just documentation for developers. It is instructions for a machine." What changes in that sentence is the reader. A component library built for human pattern recognition is a completely different artifact from one built as machine-readable instructions.

Agents parse structured text, semantic metadata, and annotated relationships. What they need from a design system is specific: component purpose (not just component name), usage rules, behavioral constraints, relationships between components, and the intent behind decisions — not visual appearance. A component named "Button/Primary/Hover" tells an agent about the visual state. It doesn't explain when to use a primary button versus a ghost button, or why a destructive action in this system always requires a secondary confirmation before the primary call-to-action appears.

The agent knows what your Button component looks like. It doesn't know when to use it.

I've seen this play out in a specific, frustrating way. A team connects an agent to a well-organized Figma library. The agent generates a settings screen. It's clean. It passes visual QA. Then a senior designer opens it and immediately spots three things: the tone-of-voice on the destructive action is too formal for this product, the empty state uses a generic icon that the system explicitly moved away from eighteen months ago, and the spacing inside the confirmation modal doesn't account for the localization expansion that German strings require. None of that is in the component library. All of it is in the heads of the designers who built it.

[Figma AI](https://www.figma.com/ai/) can apply a design system when that system is machine-readable. When it isn't, the agent doesn't stop and ask for clarification. It makes reasonable-looking guesses. And reasonable-looking guesses, generated at scale across an entire product, accumulate into something that feels almost right — which is harder to catch and correct than something obviously wrong.

## The write-access moment

![A Figma canvas viewed from above showing an AI agent’s cursor placing a component that doesn’t connect to the existing design system, highlighted by a lime green dashed border showing the gap.](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*elviQg4mMH_OQJBQIgjQ1A.png)

illustration by author | Generated by Gemini

In March 2026, [Figma opened write access to the canvas for agents](https://www.figma.com/blog/the-figma-canvas-is-now-open-to-agents/). No read access. Write access. Via the `use_figma` tool, agents like Claude Code and Codex can now create and modify design assets directly inside your files, extending the standards you've established — or ignoring them, if those standards aren't machine-readable.

This changes the stakes in a specific way. Agents were already consulting your design system when generating code. Now they're building inside it. The same file your team treats as the canonical source of truth is now being written to by systems that can only work with what's been made explicit.

[Francesca Tabor](https://www.francescatabor.com/articles/2026/3/31/building-a-figma-driven-mcp-production-pipeline) documented this in a five-step MCP production pipeline: Get Metadata → Get Screenshot → Get Code Connect Map → Get Variable Definitions → Generate code. The sequence is elegant when every layer is structured. When the Variable Definitions step returns "see the Figma file," the sequence breaks. The agent either halts or improvises.

The sequence only works if the context is there. Most design systems stop at step 2.

[Figma's own state of the Designer 2026 report](https://www.figma.com/reports/state-of-the-designer-2026/) found that only 32% of designers and developers say they can rely on AI-generated output in their work. That skepticism is reasonable. But it's pointing at the symptom rather than the cause. The outputs aren't unreliable because the models are weak. They're unreliable because the context those models are working from is incomplete.

## The Context Gap

![An iceberg diagram showing ~40% of product context visible above the waterline (code) and 60% invisible below (brand reasoning, interaction principles, rejected decisions), with a lime green line marking the gap at the waterline.](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*jCmOr1osCEQRKiuCIjM33Q.png)

illustration by author | Generated by Gemini

[Gregory Muryn-Mukha](https://uxdesign.cc/your-ai-agent-can-read-your-codebase-it-doesnt-know-your-product-b5ea0cd77989), a founding product designer at Twain, published an article in April 2026 with a claim worth sitting with: code carries maybe 40% of the product context an agent needs — enough to infer tokens, layout conventions, and API shapes. The other 60% lives outside the code.

That 60% is the Context Gap.

It lives in design files, in marketing briefs, in the reasoning behind decisions your team made during a sprint two years ago that you never wrote down. It lives in the interaction principles your senior designer articulates in reviews, but hasn't been encoded anywhere. It lives in the brand nuances that experienced designers recognize immediately and apply intuitively — the ones that make a product feel like itself rather than like a capable SaaS template.

The Context Gap isn't about missing components. It's about missing meaning.

What makes this more than a technical problem is who owns that 60%. It's the design systems practitioners who built the component library in the first place. They know what the components are for — the organizational context, the user research that shaped the interaction patterns, the decisions that were considered and rejected, the rules that are enforced,d and the ones that bend under specific circumstances. That accumulated judgment has never needed to be made explicit because it lived in the heads of experienced designers who applied it naturally.

Think about what it takes to onboard a new designer to a mature design system. You don't just hand them the Figma file. You walk them through the thinking: why the primary navigation uses icons without labels at desktop scale, what the research said about progressive disclosure in the checkout flow, and why the color-pairing rules are stricter than WCAG technically requires. That walkthrough — the reasoning layer — is what the agent is missing. The new designer gets it through conversation. The agent gets nothing.

Agents can't work from a judgment that was never written down. They work from what's in the system. And in most design systems today, the ratio is inverted: roughly 90% visual, 10% semantic. Agents need the inverse.

## Context engineering is the successor skill.

![Two versions of a design token card — the standard value-only version above, and an expanded version below with a lime green annotation block containing usage rules, connected by an arrow labeled “Context layer added.”](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*PW3U0tKIAbOCYrS98cTbhQ.png)

illustration by author | Generated by Gemini

In July 2025, [Gartner named context engineering as a core emerging discipline](https://www.gartner.com/en/articles/context-engineering), predicting it would appear in 80% of enterprise AI tools by 2028. Context engineering is the practice of designing systems that dynamically assemble the right information, tools, and institutional knowledge into an agent's working memory before it acts.

That is a design systems problem.

The skills required to do it well are exactly the skills design systems practitioners have spent years developing — just applied to a different output. Naming conventions that carry semantic meaning beyond visual description. Token annotations that include usage rules, not just values. Component descriptions that encode behavioral intent — when to use this pattern, and when not to.

Decision rationales that explain the "why" behind the "what".

Concretely: spacing-lg is a token value. "Use spacing-lg for between-section gaps, not within-component spacing, "is the context. Both belong in a machine-readable system. Only one exists in most design systems today.

A component library tells an agent what exists. Context tells it what any of it means.

[Claude Code](https://claude.ai/code) agents can parse a well-structured context layer and generate reliable, brand-consistent outputs. The capability is there. What's missing, in most organizations, is the context layer itself — because no one has been asked to build it yet. Most design system teams are still building for the human reader. They are building for the wrong audience.

I've watched this land awkwardly in conversations with systems teams. The work that needs doing is documentation that reads differently — written not for the developer who wants to understand why the accordion pattern was chosen, but for the agent that needs to know, before it generates anything, exactly when an accordion is appropriate and when it's not.

## What the teams ahead are already doing

![A code directory view showing DESIGN.md added alongside README.md, with the agent icon pointing to it, and two output screens below comparing generic vs. brand-consistent AI-generated UI.](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*TV2Cm-LFeycU23rQ3rQZ2Q.png)

illustration by author | Generated by Gemini

On April 21, 2026, [Google Labs open-sourced DESIGN.md](https://blog.google/innovation-and-ai/models-and-research/google-labs/stitch-design-md/) — a markdown specification for describing a visual identity to coding agents. You drop it in the project root the way you'd drop a README. Compatible tools, including Claude Code and Cursor, read it before generating any interface. The file combines machine-readable YAML tokens with plain-text explanations of why each decision exists.

That structure — value plus reasoning is the difference between a component library and a context layer. It's available as open source [on GitHub](https://github.com/google-labs-code/design.md), in alpha, with 11.5K stars in its first few weeks.

[Into Design Systems' 2026 conference](https://www.intodesignsystems.com/agentic-design-systems) surfaced what the teams furthest ahead are doing. At Indeed, a pipeline that auto-converts design system MDX files to JSON metadata produced 4,300 AI prototypes in four months. Spotify's Encore design system now exposes documentation directly to AI agents via an MCP server — tools like Cursor can generate code aligned to Spotify's standards without any manual guidance.

The teams ahead aren't writing more components. They're writing context: usage rules, semantic relationships, decision rationales, and behavioral contracts. They have recognized that the design system is now infrastructure for machines, and they are building it accordingly.

An agent without context doesn't fail loudly. It generates something that looks right and isn't.

## What this means for your role

![An asymmetric Venn diagram with human designer on the left, AI agent on the right, and the lime green overlap zone in the middle labeled “Context architecture” — the only region that serves both readers.](https://miro.medium.com/v2/resize:fit:1400/format:webp/1*FV1Ez-BkfAodYw02Oevamg.png)

An asymmetric Venn diagram with human designer on the left, AI agent on the right, and the lime green overlap zone in the middle labeled “Context architecture” — the only region that serves both readers.

The design systems practitioners who will matter most in agent-augmented product teams aren't the ones who documented the most components. They're the ones who understood what the system was for — and can now make that understanding machine-readable.

The skill required is an existing one applied to a new reader. The judgment that goes into building a design system human designers actually use is the same judgment required to build a context layer that AI agents can actually follow—the format changes. The reader changes. The urgency is new.

The Context Gap is real, it's measurable, and closing it is the most under-requested design systems skill of 2026. Most organizations haven't thought to ask for it yet. That won't last.

Start with one component. Write its purpose — not its visual description, its purpose. Write when to use it and when not to. Write the decision it encodes and the alternative that was rejected. Read it back as if you've never seen the product before. If it tells you what you need to know to make the right call, you've written context. If it still requires the Figma file to make sense, you haven't.

Your design system was never just for designers. The question is whether it was built for the machines that are now reading it.
